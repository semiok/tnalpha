# tnalpha 开发需求文档

> 内容生产协作平台。一句话：**定义品牌 → 多源选题 → 风格化写作 → 智能排期 → 数据反馈，反馈回流再优化选题，形成闭环。** 三种角色协作完成。
>
> 本文档面向"用 AI 辅助开发"的协作者——读完即可理解整套系统、画出原型、分模块开发。配套高保真原型在 `docs/prototype.html`。

---

## 一、这是什么 / 给谁用

文化传媒团队的**内容生产流水线工具**。把"给一个品牌做内容运营"这件事，拆成可协作、可复用、能自我优化的系统。

- **核心痛点**：选题者凭感觉选题，但"自己觉得好 ≠ 市场买单"；写作风格全靠人;发过的文章数据沉淀不下来、用不上。
- **核心价值**：用**真实发布数据**反向校准选题（闭环），用 AI 多源推荐 + 风格抓取提效，沉淀成可复用的数据资产。
- **用户**：3 种角色（见第三节）。

---

## 二、整体架构（六模块 + 闭环）

```
            ┌──────────────── 闭环回流 ──────────────────────────┐
            ↓                                                      │
①知识库 ──→ ②选题库 ──→ ③写作引擎 ──→ ④排期版 ──→ ⑤数据反馈 ─────┘
 素材底座     产出选题      生成图文       排布发布      回收效果+复盘
                                                              │
            ⑥权限管理（三角色·横切所有模块，不在流水线上）      │
                                                              ↓
        ⑤的反馈+复盘洞察 → 沉淀【全局数据源池】→ 喂回②选题推荐（闭环闭合）
```

**关键概念：两级树**。一个**品牌**（顶层，长期不变）下挂多个 **campaign**（活动，有时效）。**选题、写作风格、排期、数据反馈都以 campaign 为单位**。品牌层只做顶层定义和汇总展示。

---

## 三、角色与权限（⑥权限管理）

三角色**严格包含**：发布者 ⊂ 选题者 ⊂ 定义者（高角色拥有低角色全部权限）。

| 角色 | 账号示例 | 职责 |
|------|---------|------|
| **定义者** owner | admin | 全部权限：建品牌/campaign、维护知识库、治理全局数据源池、账号管理 |
| **选题者** editor | admin1 | 选题推荐/采纳、写作风格、生成文章、排期；知识库只读 |
| **发布者** publisher | admin2 | 发布 + 回填数据、看板/复盘查看；其余只读 |

**权限矩阵**

| 动作 | 定义者 | 选题者 | 发布者 |
|------|:---:|:---:|:---:|
| 知识库 增删改 / 上传 / 全局池治理 | ✓ | ✗ | ✗ |
| 知识库 浏览 / 下载 | ✓ | ✓ | ✓ |
| 选题 推荐/采纳/删 · 写作风格抓取 | ✓ | ✓ | ✗ |
| 排期 排布/AI推荐/改期/移除 | ✓ | ✓ | ✗ |
| 发布 + 发布回填 + 阅读数据回填 | ✓ | ✓ | ✓ |
| 看板浏览 / 生成复盘 | ✓ | ✓ | ✓（看）|
| 账号权限管理 | ✓ | ✗ | ✗ |

**双层保护**：① 界面隐藏越权按钮；② 服务端强制校验，越权请求直接拒绝（HTTP 403）。即使绕过界面也挡得住。

## 四、模块详述

> 每个模块给出：**做什么 · 数据模型 · 界面与交互（线框）· 关键规则**。线框用字符画示意结构，不代表最终视觉。

### ① 知识库

**做什么**：管理一个品牌的所有素材，分两级树存放，供下游 AI 生成时阅读。

**数据模型**
```
品牌 Brand（顶层·低时效）
├─ 品牌定义（prompt / 内容要求：这个品牌是什么调性、写什么）
├─ 品牌层资料（低时效文档·多份：历史脉络、视觉规范等）
├─ 品牌级·关联信息层（见下）
└─ Campaign（一个品牌下多个·强时效）
     ├─「品牌日常」= 每品牌默认1个常驻campaign（无起止·不过期）→ 装日常选题
     ├─ 普通 campaign：名称 / 开始日 / 结束日
     ├─ campaign 资料（扁平·每份带"备注标识"如开幕/中期，AI 自己读时效）
     ├─ campaign级·关联信息层（见下）
     └─ 选题（归属 campaign，见②）

关联信息层（品牌、每个campaign 各有一份）：参考性资料
  ├─ 触网/不触网 标记（治理）
  └─ 来源：① 自己上传  ② 从【全局数据源池】选用

【全局数据源池】平台级共享底座（独立数据层）
  = ⑤反馈/复盘自动沉淀入池 + 共享资料 + 特殊资料
  每条带 触网/不触网 标记 + 来源/品牌 tag；所有品牌的关联层都能选用
```

**界面与交互（品牌总览）**
```
品牌：敦煌IP                                  [编辑品牌定义]
品牌定义：以"感知距离"为母题，敦煌美学的当代转译…
── 品牌层资料（低时效·长期）──────────  [⬆ 上传]
   📄 历史脉络.pdf    📄 视觉规范.docx
── 品牌级·关联信息层 ──────────  [⬆上传] [＋从池选]
   🔒不触网 赞助保密协议.pdf   ✅触网 非遗白皮书〔引自池〕
── Campaign（强时效·多个）────────  [＋ 新建campaign]
   ▸ 品牌日常   常驻·无时效        选题12 · 资料3  进入→
   ▸ 敦煌当代美术展  07.01–09.30   选题8  · 资料6  进入→
   ▸ 敦煌音乐节      08.15–08.20   选题3  · 资料3  进入→
```
campaign 详情页：起止时间 + 扁平资料列表（每份带备注）+ 本 campaign 关联层 + "去选题"入口。

**关键规则**
- **选题一律归属 campaign**，不挂品牌；"品牌-不变"的日常选题放进"品牌日常"常驻 campaign。
- **触网/不触网**：🔒不触网 = AI 仅本地阅读、不外传、不进云端模型、不被用于训练。**仅作用于关联信息层 / 数据源池**。
- campaign 资料**扁平上传**（不按时间点分层），AI 自动识别时效，备注仅辅助。
- 三级内容均**定义者维护、其他角色只读**。

### ② 选题库

**做什么**：以 campaign 为单位，AI 多源推荐一批选题 → 选题者筛选出好的 → 入库待写作。

**数据模型**
```
选题 Topic（归属 campaign）
├─ 标题 / 纲要
├─ 类型：日常·低时效 | campaign·高时效
├─ 来源（可多个）：知识库 / 热点 / 小红书 / 公众号 / 网络搜索
├─ 推荐理由（AI 为什么推这条）
├─ 反馈依据（可选：参考了哪条过往数据）
├─ 关联链接（可选：相关热点/笔记）
└─ 状态：候选 →〔采纳〕→ 待写作 → 写作中 → 图文完成 →〔进排期〕
         （弃选 = 直接删）
```

**界面与交互（选题库主页·某 campaign）**
```
── 推荐控制 ──
品牌 ◀━━●━━━━━━▶ campaign    30% : 70%   （比例条，拖动调配比）
生成数量 [10]   → 3条日常 + 7条时效
启用源  ☑知识库 ☑热点 ☑小红书 ☐网络搜索 ☐公众号
☑ 参考过往发布反馈                         [✦ 推荐选题]

── 候选区（AI推荐·勾选采纳）──
☐ [日常] 敦煌色彩美学日常科普   来源:知识库
☑ [时效] 美术展开幕KOL打卡攻略  来源:热点+小红书
☑ [时效] 展品背后的修复故事     来源:公众号
                          [✓ 采纳选中(2)]  [✕ 弃选]

── 正式选题库（已采纳·待写作→完成）── 筛选:全部▾
[时效] 美术展沉浸式体验测评   待写作   [去写作][删除]
[日常] 敦煌纹样的现代设计     写作中…
[时效] 开幕首日现场速记       图文完成 [已排期]
```
点选题"详情"：看完整纲要 / 来源 / 推荐理由 / 反馈依据 / 关联链接，再决定采纳或弃选。

**关键规则**
- **比例条**：左品牌右 campaign，调"日常 vs 时效"产出配比；数量默认 10 可改，按比例配额。
- **多源勾选**：每次推荐前选启用哪些源（控成本）。**不触网资料只本地理解、绝不外发进搜索 query**。
- **参考反馈**：推荐时读本品牌/campaign 过往已发布文章的真实数据（赞藏评/阅读），倾向高表现方向——闭环在选题端发力。
- **候选区→采纳两段式**：先进候选区，勾选采纳进正式库，弃选直接删。
- 操作权限：选题者 + 定义者。

### ③ 写作引擎

**做什么**：按 campaign 抓"同类感觉"的写作风格供参考，生成文章时把风格当提示词注入。

**数据模型**
```
写作风格 Style（归属 campaign，默认抓 5 种）
├─ 名称（如"沉浸式种草体"）
├─ 风格画像：段落结构 / 语气 / 用词（AI 提炼，可注入的提示词）
├─ 参考链接（来源真实文章）+ 来源平台（小红书 / 公众号）
└─ is_default（campaign 默认风格，生成时注入）
Campaign 增字段：default_style_id
```

**界面与交互（campaign 写作风格库）**
```
来源 ☑小红书 ☑公众号 · 默认抓5种          [🔄 重新抓取风格]
⭐默认  沉浸式种草体              📕小红书
   段落:短句铺陈+场景代入 / 语气:亲切安利 / 用词:口语化
   参考:xiaohongshu.com/xxx                   [设默认✓][查看]
○      文化科普体                📰公众号
   段落:总分总+知识点 / 语气:娓娓道来 / 用词:典雅
   参考:mp.weixin.qq.com/xxx                  [设默认][查看]
○ 攻略清单体 · 故事叙事体 · 情绪共鸣体（各带画像+链接）
```
生成（在选题卡上）：
```
选题：美术展沉浸式体验测评
注入风格：⭐沉浸式种草体（campaign默认）  [临时换风格▾]
[⚡生成文章] → 文章生成中… → ✓文章 [🖼生成图片] → ✓图文完成
```

**关键规则**
- **风格归 campaign**：不同活动调性不同；AI 去小红书/公众号搜同类文章，默认提炼 5 种。
- **campaign 设默认风格**：本活动所有选题生成都注入它（一处设、处处用），可临时换。
- **与品牌定义叠加**：品牌定义管"说什么/调性底线"，写作风格管"怎么写/表达模板"，两者叠加不互相覆盖。
- **图文两步**：先生成文章、再生成配图，状态机 6 态（待生成/文章生成中/文章已生成/配图生成中/图文完成/失败）。本期生成统一走云端。
- 操作权限：选题者 + 定义者。

### ④ 排期版

**做什么**：把"图文完成"的选题排到发布日历上，AI 给默认推荐，人工可调。

**数据模型**
```
排期分两层：
【操作层】campaign 级排期板（可操作）
  ├─ 周 + 周内到天；选题增字段 publish_date
  ├─ 待排期池：本 campaign 图文完成·未排
  ├─ AI 推荐排期：选题者设频率(篇/周) → AI 按 campaign 时间窗(开幕/中期/闭幕)+选题类型排到日期 → 手动改
  └─ 发布回填（发布者）：平台 + 文章链接 + 发布时间 → 标记已发布·锁定
【展示层】品牌级综合排期表（只读·不可操作）
  └─ 汇总旗下所有 campaign 的已排/已发，看整体节奏 + 撞期预警
```

**界面与交互**
```
【campaign 排期板·操作层】
[✦ AI推荐排期]  频率[3]篇/周        [＋新增周]
── 待排期池（图文完成·未排）──  [时效]修复故事
── 第27周 06.30–07.06 ──
  07.01(开幕) [时效]开幕速记 ✓图文   [发布][改期][移除]
  07.05       [时效]看展攻略 ✅已发布·小红书 →链接 [数据反馈]

【品牌综合总览·只读】
第27周
  07.01 [美术展·时效] 开幕速记 ✅已发布
  07.05 [美术展·时效] 看展攻略 ✅已发布
  07.05 [音乐节·时效] 阵容预告 ✓图文  ⚠ 同日2篇
```
**发布回填弹框（发布者点"发布"）**
```
选题：看展攻略
平台   (•)小红书  ( )公众号  ( )其他
文章链接 [____________________]
发布时间 [2026-07-05 10:00]
                    [取消]  [确认发布]
```

**关键规则**
- **写作之后才排期**（排的是图文完成的成品）。
- 粒度：周为主、周内精确到天。
- **campaign 级操作、品牌级只读汇总**（两层与⑤看板对称）。
- 排期操作 = 选题者+；发布 + 发布回填 = 发布者+（所有角色）。

### ⑤ 数据反馈

**做什么**：回填已发布文章的真实阅读数据，AI 生成复盘洞察，沉淀回闭环。

**数据模型**
```
回填 MediaStats（每篇已发布选题）
├─ 发布信息：平台 / 链接 / 发布时间（排期板已填）
├─ 小红书：赞 / 评 / 藏
└─ 公众号：阅读 / 点赞 / 转发

复盘洞察 Insight（campaign 级·手动"生成复盘"）
├─ AI 分析本 campaign 所有已发布数据 → 洞察(高表现 类型/角度/风格/时间) + 下轮建议
└─ →（采纳）沉淀进【全局数据源池】→ 喂②选题推荐  ★闭环闭合

看板两层：campaign看板（明细+复盘） / 品牌看板（只读·跨campaign趋势排行）
```

**界面与交互（campaign 看板）**
```
── 已发布明细 ──                              [生成复盘]
篇目        平台    赞/阅  评  藏/转  发布日
开幕速记    📕小红书 340   42   210   07.01
看展攻略    📕小红书 120   18    95   07.05
展品故事    📰公众号 1.2k  --    88   07.08
── AI 复盘洞察（手动生成）──
• 高表现：故事叙事体 > 攻略清单体；开幕节点是流量峰
• 周五发布互动最高
• 下轮建议：多用故事体、时效题卡开幕/闭幕、增公众号深度
              [采纳进全局池]  [重新生成]
```
回填弹框（发布者）：平台/链接/时间只读（排期板已填）+ 填 赞/评/藏（或 阅读/点赞/转发）。

**关键规则**
- 回填阅读数据 = 发布者+（所有角色）；生成复盘/看板 = 选题者+。
- **采纳进全局池**：复盘洞察沉淀进全局数据源池（定义者治理），喂回②选题——闭环闭合。
- 两层看板与排期的两层对称。

## 五、技术选型（建议）

> 核心诉求：多个产品经理用 AI 各开一个模块，GitHub 协作合并。所以选**主流、AI 熟、PM 易懂、模块边界清晰**的栈。

**地基栈：模块化单体（一个仓，每模块一个独立包）**

| 关注点 | 选型 | 理由 |
|--------|------|------|
| Web 框架 | FastAPI（Python） | 主流、AI 最熟、上手快 |
| 数据模型/ORM | SQLModel | 类型清晰 |
| 数据库 | PostgreSQL + Alembic 迁移 | 团队多人协作；Alembic 管 schema 演进 |
| 前端 | 服务端渲染 Jinja + HTMX + Alpine.js | 低门槛、不上重型 SPA |
| 富交互 | SortableJS（拖拽排期）+ ECharts（看板图表） | 按需引入 |
| 鉴权 | 复用三角色 RBAC | 现成 |
| AI 调用层 | 统一 `core/llm`：写文章 + 配图，配置集中可换模型 | — |
| 搜索源 | `core/sources/` 适配层：**先用假数据(stub)跑通闭环，真实源(热点/小红书/公众号)按统一接口后续插入** | 源易变，先定接口契约 |
| 文件存储 | 本地磁盘起步（接口预留对象存储） | — |
| 后台任务 | 线程 + 状态轮询起步（量大再上任务队列） | 生成/抓取是慢任务 |

**仓库结构（每人认领一个 modules/ 子目录）**
```
repo/
├─ core/        鉴权 · 数据库 · AI调用 · 搜索源适配 · 存储 · 后台任务（共享底座·谨慎改）
├─ modules/
│  ├─ knowledge/ ①   ├─ topic/    ②   ├─ writing/  ③
│  ├─ schedule/  ④   └─ feedback/ ⑤   （权限在 core/auth）
├─ templates/   各模块子目录
├─ migrations/  Alembic
└─ tests/
```
**协作方式**：每人用 AI 负责一个 `modules/` 子目录；模块之间通过**清晰的数据模型 + 接口契约**对接；`core/` 是共享底座，改动要谨慎。

## 六、闭环数据流（系统怎么转起来）

1. **定义者**在①建品牌、建 campaign、上传资料。
2. **选题者**在②选某 campaign，设比例+源，AI 多源推荐 → 候选区筛选采纳。
3. **选题者**在③给 campaign 抓写作风格、设默认，给采纳的选题生成图文。
4. **选题者**在④把图文完成的选题 AI 推荐排期 + 手动调，排到发布日历。
5. **发布者**到点发布，回填平台/链接/时间；发布后回填阅读数据。
6. **选题者/定义者**在⑤生成复盘洞察，采纳进全局数据源池。
7. **回到第 2 步**：下一轮推荐时读取这些反馈+洞察，选题更准。闭环。

## 七、术语表

| 术语 | 含义 |
|------|------|
| 品牌 Brand | 顶层、长期不变的内容主体（如"敦煌IP"） |
| campaign | 品牌下的活动，有时效（如"敦煌当代美术展"）；选题/风格/排期/数据都按它走 |
| 品牌日常 | 每品牌默认1个常驻 campaign（无时效），装日常低时效选题 |
| 关联信息层 | 参考性资料层，品牌和每个 campaign 各有一份 |
| 触网/不触网 | 资料治理标记；不触网=AI 仅本地读、不外传、不进云端模型 |
| 全局数据源池 | 平台级共享数据层，沉淀反馈/复盘/共享料，各品牌可选用 |
| 候选区→采纳 | 选题两段式：AI 推荐先进候选区，人工筛选采纳进正式库 |
| 写作风格注入 | 生成文章时把 campaign 默认写作风格作为提示词加入 |
| 闭环 | 发布数据→复盘→全局池→选题推荐，用真实数据校准选题 |
